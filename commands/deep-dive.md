# /deep-dive — 도메인 빠르게 파악하기

외주 PM이 낯선 도메인에 투입됐을 때, 도메인의 실제 업무 프로세스와 복잡도를 분석하고 기획 시 놓치기 쉬운 부분을 미리 잡아주는 커맨드입니다.

## Invocation

```
/deep-dive 빵집에서 식자재 OCR로 찍어서 원가 계산하는 서비스
/deep-dive 중고차 딜러를 위한 차량 상태 검수 자동화 플랫폼
/deep-dive 병원 예약 시스템에서 진료 후 보험 청구까지 자동화
/deep-dive                          # 인터랙티브 모드 — 어떤 서비스인지 물어봄
```

## Workflow

### Step 1: 서비스 정보 파악

**인자가 있는 경우** — 바로 Step 2로 진행

**인자가 없는 경우** — 아래 질문을 순서대로:
1. "어떤 서비스를 기획하시나요? 한두 줄로 설명해주세요."
2. "타겟 사용자(고객)가 누구인가요? (예: 빵집 사장님, 병원 원무과 직원)"
3. "현재 가지고 있는 자료가 있나요? (PRD, 세일즈 자료, 기획서 등) 있으면 첨부해주세요."

사용자가 파일을 첨부하면, 먼저 읽고 서비스 컨텍스트에 반영합니다.

### Step 2: 도메인 리서치 및 분석

**domain-deep-dive** 스킬을 적용합니다:

- 웹 검색으로 해당 도메인의 실제 업무 프로세스를 조사
- 5개 섹션(As-Is 워크플로우, 복잡도 지도, Unknown Unknowns, Must-Ask 질문, 위험도 체크) 작성
- 사용자가 첨부한 자료가 있으면 자료 내용과 대조하여 이미 파악된 부분/아직 부족한 부분 구분

### Step 3: 후속 액션 제안

분석 결과를 제시한 후, 아래 후속 액션을 제안합니다:

- "질문 리스트를 노션/문서로 정리해드릴까요?"
- "특정 섹션을 더 깊이 파고 싶으시면 알려주세요."
- "이 도메인의 경쟁 서비스나 유사 서비스를 분석해드릴까요?"
- "기획 시작 전 체크리스트를 만들어드릴까요?"

## Output Format

```
# Domain Deep-Dive: [서비스명]

## 1. 이 일이 실제로 어떻게 돌아가는지
[현재 업무 흐름 + 핵심 용어 사전]

## 2. 겉보기엔 쉬운데 실제론 복잡한 것들
[숨은 복잡도 + 예외 케이스 + 규제]

## 3. 이걸 몰랐으면 나중에 이렇게 터짐
[놓치기 쉬운 포인트들 — "만약 이걸 몰랐다면" 형식]

## 4. 클라이언트한테 꼭 물어봐야 할 질문들
[카테고리별 질문 리스트]

## 5. 지금 자료로 기획하면 위험한 부분
[위험도 높음/중간/낮음 분류]

## 다음 단계 추천
[지금 바로 / 이번 주 안에 / 기획 착수 전에]
```

## Notes

- 이 스킬은 **도메인을 전혀 모르는 상태**를 전제로 합니다. 이미 도메인을 잘 아는 경우에는 `identify-assumptions` 같은 가정 검증 스킬이 더 적합합니다.
- 웹 리서치를 포함하므로 인터넷 연결이 필요합니다.
- 첨부 자료(PRD, 세일즈 자료)가 있으면 분석 품질이 훨씬 좋아집니다.
